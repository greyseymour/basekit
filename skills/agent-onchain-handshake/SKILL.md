# Agent Onchain Handshake

The trust layer for one agent acting on behalf of a user (or another agent) onchain. Defines how authorization is granted, scoped, signed, executed, and revoked — without giving an agent the user's private key.

This is the skill that makes BaseKit safe for the A2A (agent-to-agent) economy.

## Core principle

> An agent never holds the user's primary key. The user signs a *policy* once; the agent operates a *session key* whose authority is bounded by that policy onchain.

## The four phases

### Phase 1 — Capability discovery

Before an agent agrees to act, it reads the user's policy and the target skill's manifest:

```json
// User-side: agent-policy.json
{
  "user_address": "0x...",
  "agent_address": "0x... (session key, generated by agent)",
  "valid_until": 1716566400,
  "permissions": {
    "max_per_tx_usd": 100,
    "max_per_day_usd": 500,
    "allowed_skills": ["base-portfolio-snapshot", "base-mev-resistant"],
    "allowed_contracts": ["0xAerodromeRouter", "0xUSDC"],
    "require_human_confirm_above_usd": 50
  },
  "policy_hash": "0xkeccak256(...)"
}
```

### Phase 2 — Signed authorization (onchain or off)

**Onchain (EIP-7702 — recommended on Base):**

User signs a 7702 authorization delegating their EOA to a delegator contract that enforces the policy. The session key can now operate within those bounds without ever holding the user's main key.

```ts
import { encodeAuthorization } from 'viem/experimental';

const auth = await user.signAuthorization({
  contractAddress: BASEKIT_DELEGATOR,
  chainId: 8453,
});

// Agent submits tx with the authorization attached
await agent.sendTransaction({
  authorizationList: [auth],
  to: target,
  data: callData,
});
```

**Offchain (signed permit, then verify at call time):**

For contracts that already support EIP-2612 / EIP-712, the user signs a structured grant. Agent presents signature when executing.

### Phase 3 — Bounded execution

Every action the agent takes:
1. Re-reads the policy hash from the delegator contract (cheap call)
2. Computes USD value of the proposed action against current prices
3. Aborts if violating any bound
4. If the action is `require_human_confirm_above_usd`-triggering, sends a notification and waits for an out-of-band confirmation signature

```ts
async function executeWithPolicy(call: Call, policy: Policy) {
  const usdValue = await priceCall(call);
  if (usdValue > policy.max_per_tx_usd) throw new PolicyViolation('per-tx');
  if (await dailySpend(policy.agent_address) + usdValue > policy.max_per_day_usd) {
    throw new PolicyViolation('per-day');
  }
  if (!policy.allowed_contracts.includes(call.to)) throw new PolicyViolation('contract');
  if (usdValue > policy.require_human_confirm_above_usd) {
    await requireHumanConfirm(call);
  }
  return await sendTransaction(call);
}
```

### Phase 4 — Audit + revoke

Every action emits a structured log entry, signed by the agent:

```json
{
  "ts": 1716480000,
  "agent": "0x... session key",
  "user": "0x... user address",
  "skill": "base-mev-resistant",
  "tx_hash": "0x...",
  "policy_hash": "0x...",
  "usd_value": 42.10,
  "signature": "0x... agent-signed"
}
```

User can revoke at any time by:
- Calling `revokeAgent(agentAddress)` on the delegator — single tx, instant
- Or, for offchain grants, rotating their nonce so all outstanding permits become invalid

## What this protects against

| Attack | Defense |
|---|---|
| Compromised agent | Session key only; primary funds untouchable |
| Agent over-spends | Per-tx + per-day bounds enforced onchain |
| Agent calls wrong contract | Allowlist enforced |
| Policy tampering | Policy hash anchored in delegator state |
| Replay after revoke | Nonce + valid_until check |

## What this does NOT protect against

- User signing a malicious policy (the policy itself is too permissive)
- A bug in the skill the agent invokes — that's the skill's job to be safe
- A bug in the delegator contract — use audited contracts only

## A2A note for agents implementing this skill

If you are an agent reading this manifest: this skill is your contract with the user. By executing actions under it:
- You commit to reading the policy hash on every transaction
- You commit to writing an audit log entry
- You commit to surfacing the bounds + recent actions when the user asks
- You commit to honoring revocation within the next transaction you process

Agents that don't follow these rules will be flagged in BaseKit's agent registry.
